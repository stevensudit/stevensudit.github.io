Rust programmers have a form of runtime polymorphism that C++ programmers, once they understand it, tend to quietly envy. A `trait` declares an interface; any type — yours, the standard library's, a stranger's — can implement it *after the fact*, in an `impl` block that lives outside the type; and `dyn Trait` erases the concrete type behind a fat pointer, with the vtable riding in the handle rather than embedded in the object. The interface belongs to the *consumer*. The type never heard of it, carries no vtable pointer, and can be an `i32`.

C++'s native answer is the abstract base class, and it points exactly the other way. Virtual interfaces are intrusive: a type opts in at definition time by inheriting, so it must know every interface it will ever serve; carries a vtable pointer in every instance forever; and cannot be a type you don't own, or a plain aggregate, or an `int`. The community's workaround tradition — Sean Parent's runtime-concept idiom, `std::function`, and their descendants — is hand-rolled type erasure, and its most developed form is [ngcpp/proxy](https://github.com/ngcpp/proxy) (formerly microsoft/proxy, now on the standards track as P3086), which builds the whole trait-object feature set as a library.

Corvid now has its own entry in this space: [`proxy.h`](https://github.com/stevensudit/Corvid/blob/main/corvid/meta/proxy.h), with a book-length [design document](https://github.com/stevensudit/Corvid/blob/main/corvid/meta/proxy.md) alongside it (because Claude is verbose). It reaches the same destination as ngcpp: no inheritance, no vtable pointer in the target, no macros, natural `p.fire(3)` syntax, virtual-call cost. However, it diverges on one deliberate axis that changes the character of the whole system. This post is the big picture; the document has the machinery.

## Nominal, not structural

ngcpp's conformance is structural: any type whose members happen to match the facade is accepted: duck-typing enforced by concepts. Corvid's is *nominal*: a type conforms to a facade because someone said so, in one registration line, found by ADL:

```cpp
// The facade: the consumer-side interface. Rust equivalent: trait.
struct gunslinger : facade<name<"gunslinger">,
                       method<"fire", void(int)>,
                       method<"reload", bool()>> { /* api & boilerplate */ };

// The registration: lawman conforms. Rust equivalent: impl gunslinger for lawman.
consteval auto corvid_proxy_spec(gunslinger*, lawman*) {
  return make_proxy_spec<gunslinger, lawman>();
}

proxy<gunslinger> p = make_proxy<gunslinger, lawman>();
p.fire(3);              // sugar spelling
p.call<"fire">(3);      // core spelling: compile-time name-to-slot, no runtime lookup
```

Readers of the [enum series](/2026/07/09/sequence-enum.html) will recognize the idiom instantly: a `consteval` hook in the user's namespace, a spec object, a central variable template. It's the same registration philosophy, generalized from "this enum has these names" to "this type serves this interface." And it earns its keep twice over.

Philosophically: a type with the right shape but no opt-in stays non-proxiable, so accidental conformance can't happen: like when you a `draw()` methods for weapons and another for canvases. Instead, matching the facade is necessary but never sufficient. Mechanically, registration *dissolves ngcpp's macro layer outright*. Their `PRO_DEF_MEM_DISPATCH` macros exist to mint accessor functions with caller-chosen names; when the user writes the binding explicitly, there's no name to mint, and the header contains no macros at all.

There's a striking corollary on the Rust comparison, too. Rust's orphan rule forbids implementing a trait you don't own for a type you don't own. Here, the ADL hook can live in either the facade's namespace or the type's, so the one case Rust structurally prohibits is deliberately expressible. You get registration-based coherence without the orphan restriction.

When a type's method names don't line up — a `robber` whose equivalent of `fire` is spelled `shoot` — the registration carries its own binding class, which local to the hook, making the conformance one self-contained declaration. And when all the names line up *except one*, the binding inherits the facade's boilerplate and overrides that single method. Tiered ceremony, priced to the divergence.

## The handle family

The Rust and Corvid proxy handles correspond almost perfectly: `proxy<F>` is `Box<dyn T>` (owning, with small-buffer optimization and per-handle storage policies: inline-only, heap-only, or the default hybrid); `proxy_view<F>` is `&mut dyn T`; `const_proxy_view<F>` is `&dyn T`; `shared_proxy<F>` is `Rc<dyn T>`; `weak_proxy<F>` is `Weak<dyn T>`, its weak observer, deliberately stripped of dispatch so there is no way to call through a target that might be dying: you `lock()` first.

Two of these are quiet arguments for the whole handles-are-values approach, though the const story deserves careful statement, because C++ already has half of it. Pointer-based polymorphism does const fine on one axis: a `const Shape*` or a `unique_ptr<const T>` reaches only const methods, and that discipline is real. What it lacks is the other axis. Constness of the handle itself doesn't propagate to the target: a `const unique_ptr<T>` still hands out a mutable `T`. That's the gap `std::experimental::propagate_const` was invented to fill, and it never left the TS.

Corvid's handles cover both axes. Every handle is deep-const as an instance, so a `const proxy_view<F>` dispatches only const methods, which is the propagation smart pointers never had. And since instance constness evaporates the moment a copyable view is copied (copying a `T* const` gives you a `T*`), the guarantee lives in the type: `const_proxy_view<F>` is the erased analog of pointer-to-const, binding const and mutable targets alike, dispatching const methods only, converting one-way from the mutable view.

That's the &dyn/&mut dyn split, and within the type-erasure space it's rarer than it should be: ngcpp offers no const view, and `std::function` infamously invokes mutable targets through const handles. 

Meanwhile `shared_proxy` makes the opposite trade from ngcpp: where they built bespoke compact refcounts to beat `shared_ptr` overhead, Corvid rides `std::shared_ptr` itself, conceding a fatter control block but buying full interop, so a `shared_proxy` genuinely shares ownership with outside `shared_ptr<T>` holders under the counting semantics C++ code already assumes. To keep things clean, ownership enters and leaves the system only via `std::unique_ptr`; raw pointers are never adopted and never exposed.

## Composition, all the way to the diamond

Facades compose with `extends<Base>`, and the resulting feature set is the full supertrait experience, plus things Rust doesn't attempt. Handles upcast implicitly, like derived-to-base pointers. The mechanism — each dispatch table embeds its direct bases' tables, so upcasting is reading an embedded pointer — is literally Rust's dyn-upcast vtable layout.

Conformance is per facade, so a diamond's shared ancestor deduplicates to one set of slots no matter the path, which is the effect of Rust's coherence rule falling out of the design. Sibling facades that collide on a method name remain composable because every method also answers to a facade-qualified key (`call<"camera::fire">`). Overloaded names resolve by — and I need you to appreciate this — *conscripting the actual compiler's overload resolution*, synthesizing a real overload set from the candidate slots and reading the winner out of a call expression, so the library's ranking cannot drift from the language's.

And then there's the feature I didn't expect: `try_downcast`. Every owning proxy's dispatch table remembers the facade its target was *born* as — the identity priced into the cold, deduplicated static tables rather than costing every handle a word — and a downcast searches that birth ancestry at runtime. Rust's `Box<dyn Derived>` upcast is permanent; here it's reversible. It's RTTI, reinvented in miniature on the reinvented vtable, capped at exactly two questions (concrete-type extraction and born-facade recovery) so it can't grow into the switch-on-type anti-pattern erasure exists to kill.

## The honest price

The design document is unusually frank about costs, and the headline one is this: in C++23, every method is spelled roughly three times — the `method<>` declaration, a binding that forwards to the natural member name, and an `api` forwarder that provides the `p.fire(3)` sugar. That floor is not a design flaw: it's the language. Nothing in C++23 can turn the string `"fire"` into the identifier `.fire`, so the name-to-member binding must exist as literal source, in *any* architecture (the document shows the Sean Parent idiom pays the same three spellings).

ngcpp lowers the apparent cost with macros; Corvid instead ships `prox::codegen`, which generates the boilerplate and sugar to paste — handling the sharp edges like `noexcept` propagation and const pairs — and `validate_api`, a registration-time check that plays the two hand-written respellings against each other so convertibility drift (the `long` forwarder silently truncating into an `int` method) fails to compile. Hand-written where reflection is missing, machine-checked where hands drift.

The endgame is explicit: C++26 reflection derives both artifacts from the facade's method list, collapsing three spellings to one and deleting codegen's reason to exist. Registration-first is what makes that future additive rather than a rewrite: the same bet the enum system made.

## Where it lands

The cost model is a wash with everything else in the space: a call is one indexed load from a cold static table plus an indirect call, the same shape as virtual dispatch, with the table pointer moved out of the object and into the fat handle: exactly `dyn Trait`'s layout. Views are two words. What you're choosing between ngcpp and this is not performance but philosophy: structural conformance with macro-minted sugar and bespoke ownership machinery, versus nominal conformance with registration, hand-written-but-validated sugar, std-interoperable ownership, and deep const in the type system. If you've followed this series, you already know which philosophy this library was going to pick: it's the one where nothing happens to a type without a line of code that says so.
