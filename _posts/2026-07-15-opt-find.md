"It is a strange fate that we should suffer so much fear and doubt over so small a thing. Such a little thing."

Supposedly, that line's about a cursed ring of invisibility, but I'm sure it was actually about [`opt_find.h`](https://github.com/stevensudit/Corvid/blob/main/corvid/containers/core/opt_find.h), a pair of functions (with a small class behind them) that takes a C++ blemish that's as old as STL and makes it disappear, like a hobbit with a ring.

You know the ritual. Every lookup in a standard container is a three-step incantation. Here's its most compact form:

```cpp
if (auto it = m.find(key); it != m.end())
  use(it->second);
```

Find, compare against `end()`, dereference. Each step has a documented failure mode that every C++ programmer has either committed or reviewed. Compare against the *wrong container's* `end()`: trivial to do when two maps are in scope, undefined behavior, no diagnostic. Dereference without checking: undefined behavior again. Check, then get cute and use `m[key]` on the next line anyway, default-constructing a value on the miss you just detected; this one's merely a bug, not UB, which is why it survives in production for years. The iterator-pair protocol dates to the STL's design in the early '90s, when iterators-as-generalized-pointers was the unifying insight. The cost of that unification is that a *lookup*, conceptually a function returning maybe-a-value, is expressed as a half-open range test at every single call site. `end()` is an implementation detail, and it leaks into every line of code that ever asks a container a question.

The fix, once `std::optional`-style vocabulary exists, is almost embarrassingly direct:

```cpp
if (auto v = find_opt(m, key))
  use(*v);
```

One object now carries both answers: whether it was found, and where it is. There is no `end()` to compare against, wrong or otherwise, and no unchecked path that compiles differently from the checked one. The declaration-in-condition form isn't just compact; it's the shape that makes the checked usage the *easy* usage.

## What actually comes back

It would be easy to assume `find_opt` returns a raw pointer, with null standing in for "not found." It doesn't, and the difference is most of the point. The return type is [`optional_ptr`](https://github.com/stevensudit/Corvid/blob/main/corvid/containers/core/optional_ptr.h), a pointer adapter that satisfies the [NullablePointer](https://en.cppreference.com/w/cpp/named_req/NullablePointer) requirements while layering on the `std::optional` interface. It works over raw and smart pointers alike, so the same vocabulary applies whether the found thing is a plain element or something owned.

The optional side is the full kit, not a gesture. `has_value()` and `operator bool` answer the found question. `value()` returns the element by reference and throws `std::bad_optional_access` on a miss, which is genuine optional behavior rather than pointer behavior; a raw pointer's version of "throw on miss" is a segfault. And the `value_or` family is where lookups collapse into single expressions:

```cpp
auto& setting = find_opt(config, key).value_or_ptr(&defaults);
int port      = find_opt(config, "port").value_or(8080);
auto name     = find_opt(cache, id).value_or_fn([&] { return load(id); });
```

`value_or` returns by value with a supplied default (or a default-constructed one if you pass nothing). `value_or_ptr` returns by *reference*, dereferencing a fallback pointer, for when you want the found object or a canonical default without copying either. `value_or_fn` takes a callable, so the expensive fallback only runs on a miss. Each of these replaces a find-check-branch stanza at the call site with one chained call, and chaining off the return value is the intended style: the class comment states flatly that it exists to be a lightweight return type, and "if you find yourself declaring variables of this type, you're doing it wrong." It lives in an `internal` namespace to discourage exactly that.

At the same time, it subtracts the pointer behaviors you never wanted from a lookup result. `operator[]` is deleted; so are `+` and `-` in both directions. A raw pointer answers a question you asked and then volunteers to walk off the end of the container through arithmetic; `optional_ptr` refuses. Every accessor is `[[nodiscard]]`, so silently ignoring a lookup result is a warning. For raw pointers there's an implicit conversion back to the pointer type, which is what lets it behave normally in a condition and dereference with `*`, and it even has a `std::formatter`, so a lookup result drops straight into `std::format` with a sensible null rendering. Pointer where pointer semantics help, optional where they don't, and nothing where the operation is a trap.

## The dispatch underneath

For sixty-three lines, the header itself shows real judgment, and it's in the overload set. There are two `find_opt`s, selected by concept. Containers satisfying `KeyFindable`, meaning those with a member `find` over a `key_type` (the associative family), get their own `find`, preserving O(log n) or O(1) lookup. Everything else satisfying `RangeWithoutFind` gets `std::find`, the linear scan, which makes `find_opt` work uniformly across `std::vector`, `std::array`, `std::set`, `std::map`, and even raw C arrays (though not arrays decayed to pointers: no size, no search, and the concept correctly refuses rather than guessing).

The subtle case is strings. `std::string::find` exists, but it's the wrong shape: it returns an index, not an iterator, and it searches for substrings under a different semantic than element lookup. The `KeyFindable` concept keys on `T::key_type`, which strings lack, so they correctly fall through to the `std::find` path, where finding a *character* in a string works exactly like finding an element anywhere else. The supporting `it_to_ptr` helper handles both currencies, iterators compared against `end` and index-with-`-1`-sentinel, so the two conventions the standard library never reconciled get reconciled one level up.

One more default worth noticing: for keyed containers, the result targets `pair.second`, the mapped value, not the pair. That's a choice, and it's the right one, because it's what the call site wants in the overwhelming majority of lookups; the iterator ritual makes you write `->second` as a suffix tax on every use. When you genuinely want the pair, `find_opt<extract_field::key_value>` says so explicitly, and the exception verbosely announces itself instead of the rule.

And since presence-testing is the degenerate lookup, the header throws in `contains(c, k)`, defined as exactly `find_opt(c, k).has_value()`, giving every container the member function that `std::map` only grew in C++20, and that `std::vector` still doesn't have.

## The general lesson

It would be easy to undersell this pairing because nothing in it is hard. No NTTP machinery, no atomics, no constraint-subsumption duels. But that's the point worth taking away: an interface wart that generates real, recurring bugs (the mismatched-`end()` UB, the `find`-then-`operator[]` double lookup) doesn't necessarily require a proportional amount of machinery to fix. It requires noticing that the ritual *is* a wart, that `optional` semantics are the vocabulary the original design predated, and that the return type is where those semantics belong, so every call site inherits them without asking. Sixty-three lines of dispatch, one small class behind them. Most of the fixes we don't write are this size; we just stop seeing the ritual.

I've been using versions of this technique for years, and it's always been a simple way to improve code readability. I like to think of it as my tricksy functions: my precious.
