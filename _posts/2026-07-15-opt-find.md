Some headers earn a post by their depth. Today's earns it by its impact on code readability: [`opt_find.h`](https://github.com/stevensudit/Corvid/blob/main/corvid/containers/core/opt_find.h) removes a blemish that's as old as STL.

You know the ritual. Every lookup in a standard container is a three-step incantation. Here's its most compact form:

```cpp
if (auto it = m.find(key); it != m.end())
  use(it->second);
```

Find, compare against `end()`, dereference — and each step has a documented failure mode that every C++ programmer has either committed or reviewed. Compare against the *wrong container's* `end()` (trivial to do when two maps are in scope; undefined behavior, no diagnostic). Dereference without checking (undefined behavior). Check, then get cute and use `m[key]` on the next line anyway (default-constructs a value on the miss you just detected: this one's merely a bug, not UB, which is why it survives in production for years). The iterator-pair protocol dates to the STL's design in the early '90s, when iterators-as-generalized-pointers was the unifying insight; the cost of that unification is that a *lookup* — conceptually a function returning maybe-a-value — is expressed as a half-open range test at every single call site. `end()` is an implementation detail, and it leaks into every line of code that ever asks a container a question.

The fix, once `std::optional`-style vocabulary exists, is almost embarrassingly direct:

```cpp
if (auto v = find_opt(m, key))
  use(*v);
```

`find_opt` returns an `optional_ptr`: a pointer wrapped in optional semantics (`has_value`, `operator bool`, `value_or`, the works). One object carries both answers: whether it was found, and where it is. There is no `end()` to compare against, wrong or otherwise, and no unchecked path that compiles differently from the checked one. The declaration-in-condition form isn't just compact, it's the shape that makes the checked usage the *easy* usage.

## The dispatch underneath

For sixty-three lines there's real judgment in here, and it's in the overload set. There are two `find_opt`s, selected by concept: containers satisfying `KeyFindable` — those with a member `find` over a `key_type`, i.e. the associative family — get their own `find`, preserving O(log n) or O(1) lookup. Everything else satisfying `RangeWithoutFind` gets `std::find`, the linear scan, which makes `find_opt` work uniformly across `std::vector`, `std::array`, `std::set`, `std::map`, and even raw C arrays (though not arrays decayed to pointers: no size, no search, and the concept correctly refuses rather than guessing).

The subtle case is strings. `std::string::find` exists, but it's the *wrong shape*: it returns an index, not an iterator, and it searches for substrings under a different semantic than element lookup. The `KeyFindable` concept keys on `T::key_type`, which strings lack, so they correctly fall through to the `std::find` path, where finding a *character* in a string works exactly like finding an element anywhere else. The supporting `it_to_ptr` helper handles both currencies — iterators compared against `end`, and index-with-`-1`-sentinel for the find-by-position world — so the two conventions the standard library never reconciled get reconciled one level up.

One more default worth noticing: for keyed containers, the returned pointer targets `pair.second` — the mapped value — not the pair. That's a choice, and it's the right one, because it's what the call site wants in the overwhelming majority of lookups; the iterator ritual makes you write `->second` as a suffix tax on every use. When you genuinely want the pair, `find_opt<extract_field::key_value>` says so explicitly, and the exception verbosely announces itself instead of the rule.

And since presence-testing is the degenerate lookup, the header throws in `contains(c, k)` — defined as exactly `find_opt(c, k).has_value()` — giving every container the member function that `std::map` only grew in C++20, and that `std::vector` still doesn't have.

## The general lesson

It would be easy to undersell this header because nothing in it is hard. No NTTP machinery, no atomics, no constraint-subsumption duels. But that's the point worth taking away: an interface wart that generates real, recurring bugs — the mismatched-`end()` UB, the `find`-then-`operator[]` double lookup — doesn't necessarily require a proportional amount of machinery to fix. It requires noticing that the ritual *is* a wart, that `optional` semantics are the vocabulary the original design predated, and that concepts can route each container to its efficient path without the caller knowing there are paths. Sixty-three lines. Most of the fixes we don't write are this size; we just stop seeing the ritual.
